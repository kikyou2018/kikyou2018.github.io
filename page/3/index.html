迭代器模式

Posted on 2024-09-12 Edited on 2024-09-13 In STUDY

重要性程度：5
迭代器模式是一种行为型设计模式，它提供了一种方法，顺序访问聚合对象（如列表、数组、集合）中的各个元素，而不需要暴露其内部的表示。迭代器模式使得客户端可以以统一的方式遍历不同类型的聚合对象，而无需关心对象的内部结构。

  1. 核心思想：迭代器模式的核心思想是将集合的遍历行为从集合对象中分离出来，将遍历的责任交给迭代器对象。这样，客户端无需知道聚合对象的内部结构，也能以统一的方式遍历集合中的元素。
  2. 主要组成
    1. 迭代器接口Iterator
    2. 具体迭代器类
    3. 聚合对象接口
    4. 具体聚合类
  3. 代码

装饰者模式

Posted on 2024-09-12 In STUDY

重要性程度：3
动态地给一个对象增加一些额外的职责，就增加对象功能来说，装饰模式比生成子类实现更加灵活。是一种对象结构型模式。与继承关系相比，关联关系的主要优势在于不会破坏类的封装性，而且继承是一种耦合度较大的静态关系，无法在程序运行时动态扩展。

  1. 核心组件
    1. 抽象组件
    2. 具体组件
    3. 抽象饰品
    4. 具体饰品
  2. 优点：
    1. 灵活性高。装饰模式与继承关系的目的都是要扩展对象的功能，但是装饰模式可以提供比继承更多的灵活性。可以通过一种动态的方式来扩展一个对象的功能，通过配置文件可以在运行时选择不同的装饰器，从而实现不同的行为。
    2. 遵循开放封闭原则。具体构件类与具体装饰类可以独立变化，用户可以根据需要增加新的具体构件类和具体装饰类，在使用时再对其进行组合，原有代码无须改变，符合“开闭原则”
    3. 单一职责原则。
  3. 缺点：
    1. 增加复杂性
    2. 调试困难
  4. 代码

策略模式

Posted on 2024-09-12 In STUDY

策略模式定义一系列算法，将每一个算法封装起来，并让它们可以相互替换。策略模式让算法独立于使用它的客户而变化，也称为政策模式。策略模式是一种对象行为型模式。
重要性程度：4

  1. 主要组成
    1. 环境类
    2. 抽象策略类
    3. 具体策略类
      根据环境的某些特质，采用不同的策略。
  2. 优点
    1. 完美支持“开闭原则”，用户可以在不修改原有系统的基础上选择算法或行为，也可以灵活地增加新的算法或行为。
    2. 策略模式提供了可以替换继承关系的办法
    3. 使用策略模式可以避免提供使用多重条件转移语句
  3. 缺点
    1. 客户端必须知道是所有的策略类，并自行决定使用哪一个策略类。
    2. 策略模式将造成产生很多策略类，可以通过使用享元模式在一定程度上减少对象的数量。
  4. 代码
    支付策略：分为现金支付、手机支付

观察者模式

Posted on 2024-09-12 In STUDY

观察者模式定义对象间的一种一对多依赖关系，使得每当一个对象状态发生改变时，其相关依赖对象皆得到通知并被自动更新。观察者模式又叫做发布-订阅模式。是一种对象行为模式。
重要性程度：5

  1. 主要组成
    1. 抽象观察者
    2. 具体观察者
    3. 抽象目标
    4. 具体目标
  2. 优点
    1. 可以实现表示层和数据逻辑层的分离，并定义了稳定的消息更新传递机制，抽象了更新接口，使得可以有各种各样不同的表示层作为具体观察者角色。
    2. 观察者模式在观察目标和观察者之间建立一个抽象的耦合。
    3. 观察者模式支持广播通信。
    4. 观察者模式符合开闭原则
  3. 缺点
    1. 通知机制耗费时间
    2. 如果存在循环依赖，会导致出现循环调用，可能导致系统崩溃。
  4. 代码

简单工厂、工厂方法与抽象工厂

Posted on 2024-09-12 In STUDY

简单工厂模式、工厂方法模式、抽象工厂模式均属于创建型模式。

简单工厂模式

  1. 主要组成
    1. 工厂类：负责实现创建所有实例的内部逻辑，通常提供一个静态方法，根据传入的参数类型来决定初始化哪种具体类的对象。
    2. 抽象产品类：定义了产品的公共接口。
    3. 具体产品类
  2. 优点
    1. 封装对象创建。工厂类含有必要的判断逻辑，可以决定在什么时候创建哪一个产品类的实例，客户端可以免除直接创建产品对象的责任，而仅仅“消费产品”；同时客户端无需知道所创建的具体产品类的类名，只需要知道具体产品类所对应的参数即可。（是否关心，主要看main函数创建时，是否回根据用户的目的进行工厂类、创建方法、返回对象的选择）
    2. 可扩展性：通过引入配置文件，可以在不修改任何客户端代码的情况先更换和增加新的具体产品类。
  3. 缺点
    1. 不符合开放封闭原则：每增加一种产品，都要修改工厂类的实现。
    2. 不符合单一职责原则：工厂类集中了所有对象的创建逻辑，职责较重，导致维护和管理困难。
  4. 代码
    生产果汁，客户端告诉工厂是什么水果，工厂负责生产果汁。
  5. 应用
  • JDK类库
  • Java加密技术

工厂方法模式

和简单工厂模式不同，为了解决简单工厂模式中工厂的职责复杂且不符合开放封闭原则。工厂方法模式将工厂类设置为一个抽象工厂类和若干个具体工厂类，之后增加具体产品类的时候只需要增加一个对应的具体工厂类即可，不需要对抽象工厂类进行更改。更见符合单一职责原则和开放封闭原则。

  1. 主要组成
    1. 抽象产品
    2. 具体产品
    3. 抽象工厂
    4. 具体工厂
  2. 优点：
    1. 用户只需要关心所需产品对应的工厂，无需关心创建细节，无需知道具体产品类的类名。
    2. 基于工厂角色和产品角色的多态性设计时工厂方法模式的关键。
    3. 扩展性：在系统中加入新产品时，无须修改抽象工厂和抽象产品提供的接口，无须修改客户端，也无须修改其他的具体工厂和具体产品，而只要添加一个具体工厂和具体产品就可以了。
  3. 缺点
    1. 添加新产品时，需要编写新的具体产品类和具体工厂类，类的个数成对增加。
    2. 由于考虑到系统的可扩展性，需要引入抽象层，在客户端代码中均使用抽象层进行定义，增加了系统的抽象性和理解难度，且在实现时可能需要用到DOM、反射等技术，增加了系统的实现难度。
  4. 代码

抽象工厂模式

对于上面的果汁的例子，一瓶果汁可以看作是由果汁本体、瓶子、标签组成。苹果果汁和橙汁的果汁本体和标签不同，但是瓶子可能是完全相同的。为了增加代码复用，且降低一个产品内部组件之间的耦合性（比如使用新的瓶子，不用更改苹果汁或橙汁的整个类的实现）。使用抽象工厂模式可以实现上面的功能。抽象工厂模式提供一个创建一系列相关或相互依赖对象的接口，而无需指定它们具体的类。抽象工厂模式又称为Kit模式，属于创建型模式。

  1. 主要组成
    1. 抽象组件类
    2. 具体组件类
    3. 抽象产品类
    4. 具体产品类
    5. 抽象工厂类
    6. 具体工厂类
      虽然上面列出了这么的类，但是具体实现还要看是否需要对应的抽象基类。
  2. 优点
    1. 抽象工厂模式隔离了具体类的生成，使得客户端并不需要知道什么被创建。由于这种隔离，更换一个具体工厂就变得相对容易。所有的具体工厂都实现了抽象工厂定义的那些公共接口，因此只需改变具体工厂的实例，就可以在某种程度上改变整个软件系统的行为。另外，应用抽象工厂模式可以实现高内聚低耦合的设计目的。
    2. 增加新的具体工厂和产品组很方便，无需更改已有系统，符合开放封闭原则。
  3. 缺点
    1. 在添加新的产品对象时，难以扩展抽象工厂来生产新种类的产品，这是因为在抽象工厂角色中规定了所有可能被创建的产品集合，要支持新种类的产品就意味着要对该接口进行扩展，而这将涉及到对抽象工厂角色及其所有子类的修改，显然会带来较大的不便。
    2. 开闭原则的倾斜性（增加新的工厂和产品族容易，增加新的产品等级结构麻烦）。
  4. 代码

单例模式

Posted on 2024-08-15 Edited on 2024-09-12 In STUDY

单例模式的主要作用是确保程序在运行的过程中仅存在一个实例，属于创建型模式。单例模式有两种构建方式，懒汉方式和饿汉方式。前者是指全局的单例实例在第一次被使用的时候构建，后者是指全局的单例实例在类装载的时候构建。其中饿汉式单例模式是线程安全的，而懒汉式则需要进行额外的处理保证线程安全。
对于单例类，需要将其构造函数设为private，并将拷贝构造和拷贝赋值进行delete处理，以此保证仅有一个实例存在。对单例实例的访问通过类中的静态方法进行。

  • 重排序问题
    代码在执行的时候，可能会被优化。具体来说，就是 _instance = new Singleton();可能会被优化为分配内存→赋值→调用构造函数。此时可能会出现的问题是：一个线程A通过第一个if语句判断_instance已经部位nullptr，于是就返回它，但此时对象还没被完全构造完成。另一个线程B看到_instance已经不为nullptr，就使用，此时会导致程序出现不可预料的错误（如访问未初始化的成员变量）

解决方法

  1. 使用局部静态变量
  • 使用局部静态变量能够实现线程安全的原理依赖于C++11引入的局部静态变量的线程安全的初始化机制。
  1. 使用std::call_once和std::once_flag

需要注意的地方
使用volatile并不能避免C++中双重检查锁定带来的线程安全问题。但是对于JAVA可以。
在JAVA中，volatile关键字的作用不仅仅是防止编译器优化，还具有内存可见性的语义，能够确保：

  1. 可见性：一个线程对volatile变量的修改对其他宪曾立即可见。
  2. 禁止重排序：volatile保证在对变量进行写操作是，写操作之前的所有操作都已完成，并且变量赋值之后的操作不会被重排序到变量赋值之前。即，它建立了内存屏障，防止指令重排序。

而对于C++中的volatile而言，主要是告诉编译器不要优化对这个变量的访问，因为它可能会在程序外部发生变化（如硬件设备或信号处理函数）。但C++中的volatile并不提供内存可见性和防止重排序的语义，也没有内存屏障的作用。
