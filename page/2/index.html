JSON项目

Posted on 2024-09-16 Edited on 2024-09-18 In STUDY

主要组成：

  1. 命名空间
  2. 静态变量和使用函数
    • max_depth:表示JSON嵌套的最大深度，防止解析过程中出现过深的递归。
    • TYPENAMES：一个字符串数组，包含JSON支持的各种类型的名称，如“Error”，“False”，”True”, “Null”, “Number”, “String”, “Object”, “Array”
    • MinValue 和 DecimalLength：与浮点数格式化相关的常量。
    • 字符串处理函数:
      1. stringContain：检查字符串 str 中是否包含子字符串 to。
      2. stringEqualTo：比较两个字符串是否相等。
      3. stringEndWith：检查字符串 str 是否以指定的 tail 结尾。
      4. stringStartWith：检查字符串 str 是否以指定的 head 开头。
  3. 枚举 JsonType 和 Type
    • JsonType：仅包含两种类型 Object 和 Array，用于标识 JSON 中的对象和数组。
    • Type：定义了 JSON 支持的所有类型，包括 Error、False、True、Null、Number、String、Object、Array。
  4. JSON类：定义了各种操作JSON对象的方法。
    1. 私有成员变量
      • brother 和 child：用于存储当前 JSON 节点的兄弟节点和子节点。这是一种树形结构，用来表示 JSON 中嵌套的对象和数组。
      • type：记录当前 JSON 节点的类型（如对象、数组、字符串等）。
      • valueString 和 valueNumber：用于存储字符串和数字类型的值。
      • name：当前 JSON 键的名称。
    2. 构造函数
      • 构造不同类型的 Json 对象：根据 JsonType、数值类型、字符串、布尔值或 nullptr 来构造不同类型的 Json 对象。
      • 移动和复制构造函数：支持拷贝构造和移动构造，允许对象之间的复制和转移。
    3. 静态方法：负责将输入的 JSON 字符串或字符数组解析为 Json 对象。内部使用 JsonParser 类实现具体的解析操作。
    4. 访问和操作方法
      • operator[]：重载了 [] 操作符，支持通过索引或键访问 JSON 对象中的值。如果当前对象是数组，使用索引访问；如果是对象，使用键访问。
      • contains：检查对象是否包含某个键。
      • take：从对象或数组中取出指定的键或索引的值，并从原始 JSON 中移除该值。
      • add：向对象或数组中添加值。可以是键值对，也可以是单个值。
      • clear：清空当前 JSON 对象或数组中的所有元素。
      • remove：从对象中删除指定键的值，或从数组中删除指定索引的值。
    5. 类型判断方法：sError、isNull、isObject、isArray 等：判断当前 JSON 对象的类型。
    6. 转换方法：toString、toInt、toDouble、toBool、toVector**：将 Json 对象转换为字符串、整数、浮点数、布尔值或向量，适用于不同类型的 JSON 值。
    7. 序列化方法：
      • toString：将 Json 对象序列化为 JSON 字符串。
      • valueJsonToString：辅助函数，用于将 JSON 值转换为字符串格式。
      • replaceAll：用于替换字符串中的某个子串。
    8. 内部递归方法
      • extendItem：在当前 Json 对象中添加新的子元素（用于对象或数组）。
      • appendNodeToJson：在 JSON 节点的子节点链表中追加新节点。
      • find：根据索引或键查找 JSON 对象中的指定元素。
      • deleteJson：递归删除 JSON 对象中的所有子节点，防止内存泄漏。
  5. JsonParser类：用于解析JSON字符串的类
    1. 主要成员变量
      • str：待解析的字符串。
      • i：当前解析到的字符位置。
      • err：存储解析时的错误信息。
      • failed：标识解析是否失败。
    2. 核心方法
      • fail：当解析失败时调用，设置错误信息。
      • consume_whitespace：跳过字符串中的空白字符。
      • consume_comment：处理字符串中的注释。
      • get_next_token：从输入字符串中获取下一个 JSON token。
      • parse_number：解析 JSON 中的数字值。
      • parse_string：解析 JSON 中的字符串值。
      • parse_json：解析整个 JSON 结构，包括对象、数组、字符串、数字等
  6. JsonIterator类
    1. operator*：解引用当前迭代器，返回当前 Json 对象。
    2. operator++：移动到下一个兄弟节点。
    3. begin 和 end：返回开始和结束的迭代器，用于范围循环。
    4. key：返回当前 JSON 节点的键（适用于对象）。
    5. value：返回当前 JSON 节点的值。

什么是JSON

Posted on 2024-09-16 Edited on 2024-09-18 In STUDY

JSON 指的是JavaScript Object Notation JS对象表示法，是一种轻量级的文本数据交换格式。JSON使用JavaScript语法来描述数据对象，但是JSON独立于语言和平台。

JSON语法

  1. 数据在名称/值对中
  2. 数据由逗号 , 分隔
  3. 使用斜杆 \ 来转义字符
  4. 大括号 {} 保存对象
  5. 中括号 [] 保存数组，数组可以包含多个对象

JSON的两种结构

  1. 对象：大括号保存，是一个无序的名称/值对集合。
  2. 数组：中括号，保存的是值的有序集合。值value可以是双引号括起来的字符串、数值、true、false、null、对象或者数组，可以嵌套。

JSON.parser()

JSON通常用于与服务端交换数据，在接收数据时一般是字符串，可以使用JSON.parse()将数据转换为JavaScript对象。

JSON的用途

  1. 数据传输：JSON最常见的用途是通过HTTP协议传输数据，尤其在web应用中用于客户端和服务器之间的数据交换。
  2. 配置文件：许多软件或工具使用JSON格式存储配置数据。
  3. 数据存储：JSON也可以用来存储结构化数据，尤其是在NoSQL数据库中，文档格式基本上就是JSON。

自定义一个JSON解析器，应该由哪些部分组成

  1. 词法分析器：将输入的JSON字符串分解为一系列的标记Token，每个Token代表JSON语法的基本组成部分，如大括号{}、方括号{}、冒号：、逗号，以及关键字和数据类型。
    • 作用：扫描输入的JSON字符串，将其切割成有意义的标记。
    • 实现细节：可以遍历字符串，判断是否是数字、字符串、布尔值、null，以及JSON语法中的特殊字符（如：，{}）
  2. 语法分析器：语法分析器负责根据词法分析器生成的标记序列，解析成树状数据结构。该结构通常是一个由对象、数组、字符串、数字等元素组成的树。
    • 作用：将JSON标记序列组织成有层次地结构（例如对象包含键值对、数组包含多个值）
    • 实现细节：通常使用递归下降解析方法，根据JSON的语法规则，构造相应的数据结构（如树或嵌套对象）
    • 具体任务：
    1. 解析对象：以{}包裹的一组键值对
    2. 解析数组：以[]包裹的一组值
    3. 解析值，解析JSON值可以是对象、数组、字符串、数字、布尔值或null
    4. 处理嵌套：JSON允许对象和数组的嵌套，因此语法分析器需要能够处理嵌套结构。
  3. 数据结构：解析器需要一个合适的数据结构来存储解析后的JSON数据。这些数据结构一般支持递归（例如树状结构），以处理JSON的嵌套关系。
    • 对象：通常使用类似于std::map或std::unordered_map的结构，键为字符串，值可以是任意JSON支持的类型。
    • 数组：通常使用类似std::vector或std::list来存储JSON数组中的元素。
    • 基础类型：包括string（字符串），double或int（数字），bool，null
  4. 错误处理：解析过程中，需要进行错误检测，并提供相应的错误提示。
    • 语法错误：如果JSON格式不正确。
    • 类型错误：值类型不符合预期，比如JSON语法要求一个对象而实际上是数组。
  5. 序列化：解析器除了解析JSON字符串外，还需要支持将内部数据结构重新转换成JSON字符串，这个过程称为序列化。
    • 作用：将解析后的数据结构重新生成JSON字符串，以便数据可以保存或传输。
    • 实现细节：根据树状数据结构，遍历每一个节点（对象或数组），并按顺序生成相应的JSON字符串。
  6. 扩展功能：格式化输出；支持注释；流失解析。

和protobuf的区别

  1. 格式和可读性
    • JSON是一种人类可读的文本格式，它使用键值对来表示对象，结构化数据简单直观，可以直接在文本编辑器中查看和编辑。常用于网络通信和配置文件。
    • Protobuf是一种二进制格式，不适合直接查看或编辑。是为数据的高效序列化和反序列化设计的，特别是在网络通信和存储方面。Protobuf更适合机器读取和处理，不适合人类直接阅读。
  2. 序列化和传输效率
    • JSON：由于是文本格式，数据较为冗长。传输数据时，JSON文件通常比二进制文件格式大很多，序列化和反序列化速度较慢，特别是当数据量很大的时候。效率低。
    • Protobuf：是二进制格式，序列化后的数据相对更加紧凑，占用更少的带宽和存储空间。由于Protobuf对数据进行了压缩优化，序列化和反序列化的效率非常高，更适合在高性能应用场景中使用，比如实时通讯、数据密集型系统。
  3. 数据结构定义
    • JSON：不需要事先定义数据结构。，数据的结构是自描述的。通过键值对可以轻松理解每个数据的含义，非常灵活，但没有强类型约束，这会导致数据在传输过程中出现不一致或解析错误。
    • Protobuf：必须预先定义数据结构，使用.proto文件定义每个数据类型的字段和规则。这种方法提供了强类型约束，确保了数据的兼容性和一致性。如果数据结构发生变化，可以通过向Protobuf schema中添加新的字段来实现向后兼容。
  4. 数据类型支持
    • JSON：支持的类型较为基础，主要包括字符串、数字、布尔值、数组、对象和null。在更复杂的场景下，可能需要自己进行扩展（例如日期、时间、二进制数据等）。
    • Protobuf：支持丰富的数据类型，除了基本的整数、浮点数、布尔值、字符串等，还支持枚举、嵌套对象、字节数组（用于存储二进制数据）等高级数据类型。同时 Protobuf 的类型系统更严格，字段可以指定为必填或可选，并且支持默认值。

  4. 将监听socket添加到epoll进行管理

  6. 将通信socke设置为非阻塞模式。将通信通信socket对应的handleClient函数添加到任务队列中。（改进的地方：将std::bind()改为lambda表达式，并将this改为shared_from_this，以确保RedisServer对象的生命周期在多线程环境中被正确管理。）（错误：对于单例而言，不需要shared_ptr）

handleClient()函数

（这里服务端的通信socket的缓冲区每次接受的都是一条指令，假如说客户端发送的是set key1 value1.那么下面的std::vectorstd::string tokens中存储的是set、key1、value1，然后command为set，对齐进行命令解析，解析之后，将tokens发给commandParser，进行处理。）

  1. 首先创建指令队列std::queuestd::string commandQueue;
  2. 创建bool变量用来指示是否开启了事务，是否进行回滚。bool startMulti = false; bool fallback = false;
  3. 对于这个socket的缓冲区，不断读取缓冲区内容。每次从缓冲区中读取的内容存储到char buffer[2048]中，并buffer[bytesRead] = ‘\0’;（这样做的目的：处理网络通信的时候，通常需要从套接字中读取数据并进行解析。当从套接字读取数据时，读取的数据并不会自动带终止符，因此需要手动添加一个终止符来确保可以被正确的解析为字符串。）
  4. 将buffer中的内容复制到std::string receivedData中，然后使用std::istringstream构造函数接受这个字符串，创建一个可以从该字符串读取数据的输入流对象iss。之后就可以使用std::istringstream的成员函数来逐个读取字符串中的单词或数值。
  5. 将receiveData中的各个string存储到std::vectorstd::string tokens中。
  6. tokens非空。command = tokens.front(); 定义std::string responseMessage;
  7. 针对不同的命令进行不同的处理。分别进行设置responseMessage, send()
    1. quit exit: stop, sendO()
    2. multi: 已开启，回复重复开启。未开启：定义一个std::queuestd::string empty; 执行std::swap(empty, commandQueue)[这里是为了清空commandQueue]，之后回复ok
    3. exec：首先判断事务是否开启。未开启，提示未开启。已开启：且！fallback，就调用executeTransaction()执行事务。 fallback提示错误
    4. discard：将startMulti、fallback设置为false，返回ok
    5. 处理常规指令
    • 未开启事务：
      1. 解析command。commandParser = flyweightFactory->getParser(command);
      2. 解析正常。responseMessage = commandParser->parse(tokens);
      3. 发送响应消息回应客户端
    • 开启事务
      1. 添加到事务队列commandQueue中（执行startMulti的时候已经清空commandQueue），解析错误的话，就将fallback设置为false（后续进行exec的时候，会提示错误）
      2. 正常解析，将receivedData加入到commandQueue中，回复”QUEUE”。
  8. 没有数据可读时，继续循环。
  9. 将通信socket从epoll中删除
  10. 关闭clientScoket。

executeTransaction(std::queuestd::string&commandsQueue,int clientSocket)

  • std::string hkeys(const std::string& key);： 获取哈希表中的所有字段名，类似 HKEYS 命令。
  • std::string hvals(const std::string& key);： 获取哈希表中的所有值，类似 HVALS 命令。

RedisValue于RedisValueType二者之间的关系体现了类型封装和多态的设计模式。
RedisValueType是一个抽象类，定义了所有Redis支持的数据类型的基本接口。而RedisValue则是封装这些具体类型并提供统一接口的类。

  1. RedisValueType：抽象基类
  2. Value模板类：具体类型的实现template<RedisValue::Type tag, typename T>
    • value是一个模板类，用于实现具体的Redis数据类型，如STRING\ARRAY\OBJECT。Value类通过tag来表示类型，通过模板参数来存储实际的数据值。不同的Redis类型（如STRING\ARRAY\OBJECT等）都通过继承Value模板类来实现。
  3. 具体类型的定义：具体类型如RedisString，RedisList，RedisObject都继承自Value模板类，并实现相关的虚函数。
  4. RedisValue类：统一接口封装。RedisValue类作为对外接口，封装了对不同Redis类型的操作，它的内部是一个指向RedisValueType的智能指针。

注：Value为什么还要单独设置一个tag呢？

  1. 类型安全和区分：tag的设置确保每个Value对象明确知道自己属于哪种Redis数据类型，并且可以在运行时轻松区分不同的类型。
  2. 提高类型判断的效率
  3. 避免代码重复
  4. 简化派生类的实现：由于tag的存在，派生类只需要简单地继承Value并提供对应的类型标签即可。
  5. 多态机制中的类型识别：在面向对象编程中，多态机制使得我们可以通过基类指针或引用来操作不同的派生类对象。tag提供了对具体对象类型的有效识别，而无需繁琐的dynamic_cast或typeid。
  6. 扩展性：设置tag还为将来的扩展提供了便利。
  7. 隔离模板类型和Redis数据类型：tag可以看作是将Redis的类型系统和模板的类型系统分离开来。模板T表示的是实际的底层数据类型，而tag表示Redis数据类型。这种设计使得一个模板类可以处理多个不同的数据类型，同时在逻辑上仍然能保持与Redis数据类型的一一对应关系。

注：不使用成员变量而使用模板参数

  1. 性能优化，使用模板参数将tag作为静态常量
  2. 编译时类型安全
  3. 模板特化的灵活性

责任链模式

Posted on 2024-09-12 In STUDY

重要程度：3
责任链模式是一种行为设计模式，它允许多个对象有机会处理请求，从而避免请求的发送者和接受者之间的耦合关系。这个模式将这些对象连接成一条链，并沿着这条链传递请求，直到有一个对象处理它为止。

  1. 主要组成
    1. 抽象处理者
    2. 具体处理者
    3. 客户端
  2. 典型应用：日志系统。定义一个不同级别的日志信息，如调试、信息、警告、错误等。
  3. 优点：
    1. 降低耦合度：请求发送者和请求接收者解耦，客户端不需要了解处理请求的具体对象。
    2. 增强灵活性：可以通过添加或删除处理者来修改链的结构。
    3. 职责单一原则：每个处理者只专注于自己能够处理的请求。
  4. 缺点：
    1. 请求未必被处理：如果链尾的处理者也无法处理请求，那么请求可能会被丢弃。
    2. 调试复杂：链的结构在运行时才确定，如果链过长，可能会增加调试的难度。
  5. 代码

代理模式

某些情况下，一个客户不想或者不能直接引用一个对象，此时可以通过一个称为“代理”的第三者实现间接引用。代理对象可以在客户端和目标对象之间起到中介的作用，并且可以通过代理对象去掉客户不能看到的内容和服务或者添加客户需要的额外服务。主要目的是控制对对象的访问（涉及到对方法的重定向、访问控制权限、延迟加载等）

  1. 主要组成
    1. 抽象主题
    2. 代理主题
    3. 真实主题
  2. 优点
    1. 控制对象的访问
    2. 延迟对象的初始化：虚拟代理可以延迟创建和初始化资源密集型对象，优化性能。
    3. 保护真实对象
  3. 缺点
    1. 增加复杂性
    2. 潜在的性能开销
  4. 应用
    1. 远程代理：用于访问位于远程服务器上的对象，通过代理对象屏蔽网络通信的复杂性。
    2. 虚拟代理：用于优化系统性能，延迟加载或初始化较大的对象。
    3. 保护代理：用于控制对对象的访问权限，例如根据用户角色或权限控制对象的使用。
    4. 智能引用代理：用于对对象的引用增加一些智能操作，比如计数、日志记录、引用跟踪等。C++11的智能指针
  5. 代码
    图像加载类，图片加载很耗时，使用代理延迟加载图片，知道需要时才加载。

外观模式

Posted on 2024-09-12 Edited on 2024-09-13 In STUDY

重要性程度：5
外观模式：外部与一个子系统的通信必须通过一个统一的外观对象进行，为子系统中的一组接口提供一个一致的界面，外观模式定义了一个高层接口，这个接口使得这一子系统更加容易使用。是一种对象结构型模式。

  1. 主要组成
    1. 外观角色
    2. 子系统角色
  2. 优点：
    1. 简化使用
    2. 减少耦合：客户端和子系统之间的耦合度降低，子系统的修改不会影响客户端
    3. 便于维护。子系统内部的变化不会影响到外观类和客户端之间的接口。
    4. 分层解耦。外观模式可以划分系统的不同部分，使系统层次更加清晰。
  3. 缺点：
    1. 过度封装
    2. 不符合开闭原则：如果要扩展外观类的功能，可能需要修改外观类的代码
  4. 代码
    家庭影院系统，子系统：DVD播放器、音响、投影仪。使用外观模式封装这些复杂的系统，提供一个简单的接口让用户操作。

适配器模式

Posted on 2024-09-12 Edited on 2024-09-13 In STUDY

重要性程度：4
适配器模式是一种结构型设计模式，它的主要目的是将不兼容的接口转换为兼容的接口，使得原本由于接口不兼容而不能一起工作的类能够协同工作。适配器模式通过引入一个适配器类，将已有类的接口适配成目标接口，从而解决类之间接口不兼容的问题。

  1. 应用场景
    1. 已有类的接口与需求不兼容：需要复用一个现有的类，但该类的接口与我们正在开发的系统接口不一致。
    2. 希望复用现有类并与新的接口集成：不想修改现有的代码，但希望能够在新系统中使用。
    3. 将一个旧的接口转换为新的接口：需要与遗留代码或者第三方库兼容，并且希望将它们的接口转换为符合系统的新接口。
  2. 适配器模式的两种类型
    1. 对象适配器：通过组合的方式，即在适配器类中持有被适配对象的实例，从而通过适配器类调用被适配对象的方法。
    2. 类适配器：通过继承的方式，即适配器类同时继承自被适配类和目标接口。C++ 中由于不支持多继承，所以通常使用对象适配器。
  3. 主要组成
    1. 目标抽象类target：客户希望调用的接口。适配器会将适配者的接口转换为target接口
    2. 适配器类adapter
    3. 适配者类adaptee
    4. 客户类
  4. 优点：
    1. 解耦调用者和接收者
    2. 支持撤销与重做
    3. 支持扩展性
  5. 缺点：
    1. 类的数量增加
    2. 命令的存储开销
  6. 使用场景
    1. 撤销和重做功能：例如文本编辑器、图像处理软件等需要撤销和重做操作的场景。
    2. 任务队列：可以将命令对象放入任务队列，按顺序执行，适用于多线程任务管理。
    3. 日志记录：命令模式可以记录命令的执行历史，用于系统恢复、回滚等操作。
    4. 宏命令：可以将多个命令组合成一个宏命令，一次性执行一组命令。
  7. 代码

命令模式是一种型卫星设计模式，将请求封装为一个对象，从而可以使用不同的请求对客户端进行参数化。命令模式允许将请求的发出者和执行者解耦，并且可以通过存储、排队、撤销、重做等操作来灵活处理命令。核心思想是将请求或操作封装为对象，通过这种方式，可以将不同的请求抽象出来，使得系统可以在运行时决定哪些命令需要执行，并且可以记录下这些命令用于撤销、重做等操作。

    2. 任务队列
    3. 宏命令
    4. 请求的参数化
  2. 主要组成
    1. 抽象命令
    2. 具体命令
    3. 接收者
    4. 调用者
    5. 客户端
