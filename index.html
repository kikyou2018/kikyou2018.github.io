STUDY 2024/9/13

服务端处理流程为

  1. 创建监听socket
  2. 设置监听的最大挂起连接的队列长度
  3. 将监听socket设置为非阻塞模式
  4. 将监听socket添加到epoll进行管理
  5. 调用epoll.wait()。如果是监听事件（fd == serverSocket）
  6. 将通信socke设置为非阻塞模式。将通信通信socket对应的handleClient函数添加到任务队列中。（改进的地方：将std::bind()改为lambda表达式，并将this改为shared_from_this，以确保RedisServer对象的生命周期在多线程环境中被正确管理。）（错误：对于单例而言，不需要shared_ptr）

handleClient()函数

（这里服务端的通信socket的缓冲区每次接受的都是一条指令，假如说客户端发送的是set key1 value1.那么下面的std::vectorstd::string tokens中存储的是set、key1、value1，然后command为set，对齐进行命令解析，解析之后，将tokens发给commandParser，进行处理。）

  1. 首先创建指令队列std::queuestd::string commandQueue;
  2. 创建bool变量用来指示是否开启了事务，是否进行回滚。bool startMulti = false; bool fallback = false;
  3. 对于这个socket的缓冲区，不断读取缓冲区内容。每次从缓冲区中读取的内容存储到char buffer[2048]中，并buffer[bytesRead] = ‘\0’;（这样做的目的：处理网络通信的时候，通常需要从套接字中读取数据并进行解析。当从套接字读取数据时，读取的数据并不会自动带终止符，因此需要手动添加一个终止符来确保可以被正确的解析为字符串。）
  4. 将buffer中的内容复制到std::string receivedData中，然后使用std::istringstream构造函数接受这个字符串，创建一个可以从该字符串读取数据的输入流对象iss。之后就可以使用std::istringstream的成员函数来逐个读取字符串中的单词或数值。
  5. 将receiveData中的各个string存储到std::vectorstd::string tokens中。
  6. tokens非空。command = tokens.front(); 定义std::string responseMessage;
  7. 针对不同的命令进行不同的处理。分别进行设置responseMessage, send()
    1. quit exit: stop, sendO()
    2. multi: 已开启，回复重复开启。未开启：定义一个std::queuestd::string empty; 执行std::swap(empty, commandQueue)[这里是为了清空commandQueue]，之后回复ok
    3. exec：首先判断事务是否开启。未开启，提示未开启。已开启：且！fallback，就调用executeTransaction()执行事务。 fallback提示错误
    4. discard：将startMulti、fallback设置为false，返回ok
    5. 处理常规指令
    • 未开启事务：
      1. 解析command。commandParser = flyweightFactory->getParser(command);
      2. 解析正常。responseMessage = commandParser->parse(tokens);
      3. 发送响应消息回应客户端
    • 开启事务
      1. 添加到事务队列commandQueue中（执行startMulti的时候已经清空commandQueue），解析错误的话，就将fallback设置为false（后续进行exec的时候，会提示错误）
      2. 正常解析，将receivedData加入到commandQueue中，回复”QUEUE”。
  8. 没有数据可读时，继续循环。
  9. 将通信socket从epoll中删除
  10. 关闭clientScoket。

executeTransaction(std::queuestd::string&commandsQueue,int clientSocket)

责任链模式
STUDY 2024/9/12

重要程度：3
责任链模式是一种行为设计模式，它允许多个对象有机会处理请求，从而避免请求的发送者和接受者之间的耦合关系。这个模式将这些对象连接成一条链，并沿着这条链传递请求，直到有一个对象处理它为止。

  1. 主要组成
    1. 抽象处理者
    2. 具体处理者
    3. 客户端
  2. 典型应用：日志系统。定义一个不同级别的日志信息，如调试、信息、警告、错误等。
  3. 优点：
    1. 降低耦合度：请求发送者和请求接收者解耦，客户端不需要了解处理请求的具体对象。
    2. 增强灵活性：可以通过添加或删除处理者来修改链的结构。
    3. 职责单一原则：每个处理者只专注于自己能够处理的请求。
  4. 缺点：
    1. 请求未必被处理：如果链尾的处理者也无法处理请求，那么请求可能会被丢弃。
    2. 调试复杂：链的结构在运行时才确定，如果链过长，可能会增加调试的难度。
  5. 代码
#include <iostream>
#include <string>

// 抽象处理者类
class Logger {
protected:
    int level;
    Logger* nextLogger;

public:
    Logger(int lvl) : level(lvl), nextLogger(nullptr) {}

    void setNextLogger(Logger* next) {
        nextLogger = next;
    }

    void logMessage(int lvl, const std::string& message) {
        if (this->level <= lvl) {
            write(message);
        }
        if (nextLogger != nullptr) {
            nextLogger->logMessage(lvl, message);
        }
    }

    virtual void write(const std::string& message) = 0; // 纯虚函数，子类必须实现
};

// 具体处理者类：DebugLogger
class DebugLogger : public Logger {
public:
    DebugLogger(int lvl) : Logger(lvl) {}

    void write(const std::string& message) override {
        std::cout << "Debug Logger: " << message << std::endl;
    }
};

// 具体处理者类：InfoLogger
class InfoLogger : public Logger {
public:
    InfoLogger(int lvl) : Logger(lvl) {}

    void write(const std::string& message) override {
        std::cout << "Info Logger: " << message << std::endl;
    }
};

// 具体处理者类：WarningLogger
class WarningLogger : public Logger {
public:
    WarningLogger(int lvl) : Logger(lvl) {}

    void write(const std::string& message) override {
        std::cout << "Warning Logger: " << message << std::endl;
    }
};

// 具体处理者类：ErrorLogger
class ErrorLogger : public Logger {
public:
    ErrorLogger(int lvl) : Logger(lvl) {}

    void write(const std::string& message) override {
        std::cout << "Error Logger: " << message << std::endl;
    }
};

// 客户端代码
int main() {
    // 创建具体的日志处理者
    Logger* errorLogger = new ErrorLogger(3);
    Logger* warningLogger = new WarningLogger(2);
    Logger* infoLogger = new InfoLogger(1);
    Logger* debugLogger = new DebugLogger(0);

    // 设置责任链：Debug -> Info -> Warning -> Error
    debugLogger->setNextLogger(infoLogger);
    infoLogger->setNextLogger(warningLogger);
    warningLogger->setNextLogger(errorLogger);

    // 测试责任链
    std::cout << "Sending a debug level message:" << std::endl;
    debugLogger->logMessage(0, "This is a debug message.");

    std::cout << "Sending an info level message:" << std::endl;
    debugLogger->logMessage(1, "This is an info message.");

    std::cout << "Sending a warning level message:" << std::endl;
    debugLogger->logMessage(2, "This is a warning message.");

    std::cout << "Sending an error level message:" << std::endl;
    debugLogger->logMessage(3, "This is an error message.");

    // 释放内存
    delete debugLogger;
    delete infoLogger;
    delete warningLogger;
    delete errorLogger;

    return 0;
}

代理模式
STUDY 2024/9/12

重要性程度：4
某些情况下，一个客户不想或者不能直接引用一个对象，此时可以通过一个称为“代理”的第三者实现间接引用。代理对象可以在客户端和目标对象之间起到中介的作用，并且可以通过代理对象去掉客户不能看到的内容和服务或者添加客户需要的额外服务。主要目的是控制对对象的访问（涉及到对方法的重定向、访问控制权限、延迟加载等）

  1. 主要组成
    1. 抽象主题
    2. 代理主题
    3. 真实主题
  2. 优点
    1. 控制对象的访问
    2. 延迟对象的初始化：虚拟代理可以延迟创建和初始化资源密集型对象，优化性能。
    3. 保护真实对象
  3. 缺点
    1. 增加复杂性
    2. 潜在的性能开销
  4. 应用
    1. 远程代理：用于访问位于远程服务器上的对象，通过代理对象屏蔽网络通信的复杂性。
    2. 虚拟代理：用于优化系统性能，延迟加载或初始化较大的对象。
    3. 保护代理：用于控制对对象的访问权限，例如根据用户角色或权限控制对象的使用。
    4. 智能引用代理：用于对对象的引用增加一些智能操作，比如计数、日志记录、引用跟踪等。C++11的智能指针
  5. 代码
    图像加载类，图片加载很耗时，使用代理延迟加载图片，知道需要时才加载。
#include <iostream>
#include <string>

// 抽象主题类
class Image {
public:
    virtual void display() = 0;  // 显示图片
    virtual ~Image() = default;
};

// 真实主题类，负责加载并显示图片
class RealImage : public Image {
private:
    std::string filename;

public:
    RealImage(const std::string& filename) : filename(filename) {
        loadFromDisk();
    }

    void display() override {
        std::cout << "Displaying " << filename << "\n";
    }

    // 模拟从磁盘加载图片的操作
    void loadFromDisk() {
        std::cout << "Loading " << filename << " from disk...\n";
    }
};

// 代理类，控制对真实图片对象的访问
class ProxyImage : public Image {
private:
    RealImage* realImage = nullptr;
    std::string filename;

public:
    ProxyImage(const std::string& filename) : filename(filename) {}

    // 代理类的 display 方法，只有在需要时才创建 RealImage
    void display() override {
        if (realImage == nullptr) {
            realImage = new RealImage(filename);  // 延迟加载
        }
        realImage->display();
    }

    ~ProxyImage() {
        delete realImage;  // 释放真实对象
    }
};

// 客户端代码
int main() {
    Image* image = new ProxyImage("test_image.jpg");

    // 图片只在真正需要时才加载
    std::cout << "Image will be displayed the first time:\n";
    image->display();  // 第一次调用时加载图片

    std::cout << "\nImage will be displayed again:\n";
    image->display();  // 第二次调用时不再加载

    delete image;  // 清理
    return 0;
}

外观模式
STUDY 2024/9/12

重要性程度：5
外观模式：外部与一个子系统的通信必须通过一个统一的外观对象进行，为子系统中的一组接口提供一个一致的界面，外观模式定义了一个高层接口，这个接口使得这一子系统更加容易使用。是一种对象结构型模式。

  1. 主要组成
    1. 外观角色
    2. 子系统角色
  2. 优点：
    1. 简化使用
    2. 减少耦合：客户端和子系统之间的耦合度降低，子系统的修改不会影响客户端
    3. 便于维护。子系统内部的变化不会影响到外观类和客户端之间的接口。
    4. 分层解耦。外观模式可以划分系统的不同部分，使系统层次更加清晰。
  3. 缺点：
    1. 过度封装
    2. 不符合开闭原则：如果要扩展外观类的功能，可能需要修改外观类的代码
  4. 代码
    家庭影院系统，子系统：DVD播放器、音响、投影仪。使用外观模式封装这些复杂的系统，提供一个简单的接口让用户操作。

适配器模式
STUDY 2024/9/12

重要性程度：4
适配器模式是一种结构型设计模式，它的主要目的是将不兼容的接口转换为兼容的接口，使得原本由于接口不兼容而不能一起工作的类能够协同工作。适配器模式通过引入一个适配器类，将已有类的接口适配成目标接口，从而解决类之间接口不兼容的问题。

  1. 应用场景
    1. 已有类的接口与需求不兼容：需要复用一个现有的类，但该类的接口与我们正在开发的系统接口不一致。
    2. 希望复用现有类并与新的接口集成：不想修改现有的代码，但希望能够在新系统中使用。
    3. 将一个旧的接口转换为新的接口：需要与遗留代码或者第三方库兼容，并且希望将它们的接口转换为符合系统的新接口。
  2. 适配器模式的两种类型
    1. 对象适配器：通过组合的方式，即在适配器类中持有被适配对象的实例，从而通过适配器类调用被适配对象的方法。
    2. 类适配器：通过继承的方式，即适配器类同时继承自被适配类和目标接口。C++ 中由于不支持多继承，所以通常使用对象适配器。
  3. 主要组成
    1. 目标抽象类target：客户希望调用的接口。适配器会将适配者的接口转换为target接口
    2. 适配器类adapter
    3. 适配者类adaptee
    4. 客户类
  4. 优点：
    1. 解耦调用者和接收者
    2. 支持撤销与重做
    3. 支持扩展性
  5. 缺点：
    1. 类的数量增加
    2. 命令的存储开销
  6. 使用场景
    1. 撤销和重做功能：例如文本编辑器、图像处理软件等需要撤销和重做操作的场景。
    2. 任务队列：可以将命令对象放入任务队列，按顺序执行，适用于多线程任务管理。
    3. 日志记录：命令模式可以记录命令的执行历史，用于系统恢复、回滚等操作。
    4. 宏命令：可以将多个命令组合成一个宏命令，一次性执行一组命令。
  7. 代码
#include <iostream>
#include <string>

// 目标接口
class Target {
public:
    virtual void request() const {
        std::cout <<  "Target: Default request.\n";
    }
    virtual ~Target() = default;
};

// 适配者类，已有的接口（不兼容）
class Adaptee {

    }

// 适配器类
class Adapter : public Target{
private:
    Adaptee* adaptee;
public:
    Adapter(Adaptee* adaptee) : adaptee(adaptee) {}
    void request() const override {
        std::cout << "Adapter: Translating request to adaptee's specific request.\n";
        adaptee->specificRequest(); // 转换调用
    }
};

void clientcode(const Target* target) {
    std::cout << "Client: I can work just fine with Target objects.\n";
    target->request();
}

命令模式
STUDY 2024/9/12

重要性程度：4
命令模式是一种型卫星设计模式，将请求封装为一个对象，从而可以使用不同的请求对客户端进行参数化。命令模式允许将请求的发出者和执行者解耦，并且可以通过存储、排队、撤销、重做等操作来灵活处理命令。核心思想是将请求或操作封装为对象，通过这种方式，可以将不同的请求抽象出来，使得系统可以在运行时决定哪些命令需要执行，并且可以记录下这些命令用于撤销、重做等操作。

  1. 命令模式的典型使用场景：
    1. 撤销与重做功能
    2. 任务队列
    3. 宏命令
    4. 请求的参数化
  2. 主要组成
    1. 抽象命令
    2. 具体命令
    3. 接收者
    4. 调用者
    5. 客户端
  3. 代码
#include <iostream>
#include <memory>
#include <vector>
#include <stack>

// 接受者类，负责命令执行
class Light {
public:
    void on() { std::cout << "The light is on\n"; }
    void off() { std::cout << "The light is off\n"; }
};

// 抽象命令类
class Command {
public :
    virtual void execute() = 0; // 执行命令
    virtual void undo() = 0;	// 撤销命令
    virtual ~Command() = default;
};

// 具体命令
class LightOnCommand : public Command {
private:
    Light& light;
public:
    LightOnCommand(Light& light) : light(light) {}
    void execute() override { light.on(); }
    void undo() override { light.off(); }
};

// 具体命令类，负责关闭灯的操作
class LightOffCommand : public Command {
private:
    Light& light;
public:
    LightOffCommand(Light& light) : light(light) {}
    void execute() override {light.off();}  // 执行关闭灯的操作
    void undo() override { light.on(); }// 撤销关闭灯，实际上就是开灯
};

// 调用者类
class RemoteControl {
private:
    std::stack<std::unique_ptr<Command>> commandHistory; // 命令历史，用于撤销
    std::unique_ptr<Command> command;
public:
    void setCommand(std::unique_ptr<Command> cmd) {
        command = std::move(cmd);
    }
    void pressButton() {
        if (command) {
            command->execute();
            commandHistory.push(std::move(command));	// 保存执行过的命令
        }
    }
    void pressUndo() {
        if (!commandHistory.empty()) {
            commandHistory.top()->undo();
            commandHistory.pop();
        }
    }
};
// 客户端代码
int main() {
    Light light;	// 接收者
    // 创建具体命令
    auto lightOn = std::make_unique<LightOnCommand>(light);
    auto lightOff = std::make_unique<LightOffCommand>(light);
    // 创建遥控器
    RemoteControl remote;
    // 设置并执行打开灯命令
    remote.setCommand(std::move(lightOn));
    remote.pressButton();
    // 设置并执行关闭灯命令
    remote.setCommand(std::move(lightOff));
    remote.pressButton();
    // 撤销上一步的操作（关灯 -> 开灯）
    remote.pressUndo();
    return 0;
}
